Common antipatterns and how to fix them

The following is a Piazza post I made to my wonderful students in my Gateway Java class. I figured these might help others as well, so on the internet they go. This guide is meant for beginners, or people who are just starting to learn to code.

As the end of the semester is approaching, I wanted to share some cool tips on writing clean code. These tips don’t apply as much to the scope of this class, but as you guys move on to much more intensive classes (Data structures, intermediate programming, …) some of these practices will help you tremendously.

I will present these tips in forms of “antipatterns”, or bad habits, and their alternatives.

Antipattern #1: Redundant boolean expressions

Let’s say we’re writing validation code to test whether a given username is between 1-10 characters or not. You might be tempted to write the code like this:

public static boolean isValidUsername(String username) {
	if (username.length() >= 1 && username.length() <= 10) {
		return true;
	} else {
		return false;
	}
}

While this is perfectly fine, there is a much cleaner, simpler way to write this piece of code. Notice that:

username.length() >= 1 && username.length() <= 10

is a boolean itself, that is, the entire expression is either true or false. Therefore, the above method can be written like this:

public static boolean isValidUsername(String username) {
	return username.length() >= 1 && username.length() <= 10;
}

You might be confused; why is this a thing? The comparison isn’t inside a if statement! Well, let’s give an example. Let’s say the username I entered is “kevin”. In that case, username.length() >= 1 gets evaluated to true, and username.length() <= 10 also gets evaluated to true. And to finally evaluate the whole expression, true && true is true. Thus the method returns true.

This kind of expression can be generalized: remember that an if statement, in the end, only evaluates a boolean. That is, an if statement will evaluate it’s content if its of the form:

if (true) { ... }

Knowing this, you can also avoid the antipattern of writing code like this:

if (isValid(username) == true) { ... }

and simply write it like this:

if (isValid(username)) { ... }

Antipattern #2: Unnecessarily nesting conditionals (important, especially if you are always getting cyclomatic complexity errors!)

Let’s say you’re evaluating some complicated conditional statement. For the sake of this example, I will be using letters (A,B,C, and D) to represent boolean expressions.

You might run into code that is written like this:

if (A) {
  if (B) {
    if (C) {
      // stuff 1
    } else if (D) {
      // stuff 2
    }
  }
}

Note that this if statement actually has only 2 conditions that are being executed. So, why don’t we write it more like it’s two conditions, like this:

if (A && B && C) {
  // stuff 1
} else if ( A && B && D) {
	// stuff 2
}

of course, if you really hate redundant code (or want to save computation power!), you can also write it like this:

if (A && B) {
	if (C) {
		// stuff 1
	} else if (D) {
		// stuff 2
	}
}

The point is, cyclomatic complexity can almost always be reduced with general steps like these.

Antipattern #3: Using large if/else or switch blocks when they aren’t necessary

Let’s say that I’m making a Fruit class, but because I want to make sure there are a standard set of fruit, I want to create a mapping between fruits and numbers. More specifically, I want to represent my fruits as integers like this:

0: Apple

1: Banana

2: Cherry

3: Date

4: Elderberry

5: Fig

6: Grape

One simple way to accomplish this is write code like this:

public String getFruitName(int fruit) {
	switch (fruit) {
		case 0:
			return "Apple";
			break;
		case 1:
			return "Banana";
			break;
		case 2:
			return "Cherry";
			break;
		case 3:
			return "Date";
			break;
		case 4:
			return "Elderberry";
			break;
		case 5:
			return "Fig";
			break;
		case 6:
			return "Grape":
			break;
	}
}

But this code is long and unpleasing to read. Also, if I told you that there are not just 6, but thousands, or maybe millions of fruit, how would you handle it with this switch statement? That’s pretty unrealistic. A more elegant way to achieve this is to do something along the lines of this:

public String getFruitName(int fruit) {
	String[] fruits = { "Apple", "Banana", "Cherry", "Date", "Elderberry", "Fig", "Grape" };
	return fruits[fruit];
}

Notice how one method is 25 lines, while the other is 4? And the latter is much more readable and maintainable that the former. When we map to an integer, make sure you use the properties of integers to your advantage! In this instance, we use the fact that integers can act as indices to an array, which allows us to quickly get the mapped fruit without needing a huge if/else or switch block. On that note:

Antipattern #4: Not taking advantage of various data representations

This is a more general case of the one before, but I’ll give an example that is especially poignant. When you write classes in Java, you will often run into having to write your own versions of the equals and compareTo methods (as we did in Project 5A!). Let’s say I want to write a compareTo method for my Fruit class above. Let’s assume that Grape is the best fruit, while Apple is the worst. Since we already have fruits mapped as integers, this should be rather easy. A compareTo method could be written like this:

public int compareTo(Fruit otherFruit) {
	int compare = 0;
	if (this.fruit > otherFruit.fruit) {
		compare = 1;
	} else if (this.fruit < otherFruit.fruit) {
		compare = -1;
	}
	return compare;
}

While this is okay, there is a slightly way to write this:

public int compareTo(Fruit otherFruit) {
	if (this.fruitVal > otherFruit.fruitVal) {
		return 1;
	} else if (this.fruitVal < otherFruit.fruitVal) {
		return -1;
	}
	return 0;
}

The main difference being that one returns the value (early) and doesn’t store an intermediate “compare” variable, while the other doesn’t.

But there is an even better way to write this. Note that most compareTo methods only require that a number is either 0, less than 0, or greater than 0. By taking full advantage of this fact, we can actually convert this entire method into a one-liner:

public int compareTo(Fruit otherFruit) {
	return this.fruitVal - otherFruit.fruitVal;
}

Note how this will return a positive value when this fruit is greater, a negative number if this fruit is smaller, and 0 if they are the same.

Antipattern #5: Not reusing code

This one is perhaps the most important one that people often miss. If you find yourself rewriting the same (or similar) code again, then you need to step back and ask yourself; “Can I turn this into a helper method?” That is probably the most important takeaway for you if you have read up to this point.

Of course there are so many more things that I could mention, but these are the main things that I’ve seen people do. I hope this brief guide helps, but otherwise, thanks so much for reading! Feel free to reach out to me if you have any questions.
